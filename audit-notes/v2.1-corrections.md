## Correction 4: SEC-001 Token Version Field Name

**Verdict Statement:** Add tokenVersion check mirroring appUsers.ts ~229.

**Correction:** The actual pattern in appUsers.ts is at approximately line 155-160 (not 229 — line numbers shifted from earlier edits). The JWT field is `tv` (not `tokenVersion`), and the DB column is `tokenVersion` on the `appUsers` table. The check is: `if (jwt.tv && jwt.tv !== user.tokenVersion) → reject`. This was implemented correctly in `authenticateDimeRequest`.

## Correction 5: SEC-002 evt_test_ Preservation

**Verdict Statement:** "PRESERVE evt_test_ handling" when changing failure branches to HTTP 400.

**Correction:** The evt_test_ check occurs AFTER signature verification succeeds (it's inside the try block that calls `stripe.webhooks.constructEvent()`). The 3 failure branches that were changed to 400 are all BEFORE or OUTSIDE the successful verification path. Therefore, evt_test_ handling is structurally unaffected — it only fires when Stripe sends a test event with a valid signature, which still returns 200 with `{ verified: true }`.

## Correction 7: BE-006 Rate Limiter Specificity

**Verdict Statement:** "Move /api/db-status + /api/perf behind globalApiLimiter + owner auth."

**Correction:** `globalApiLimiter` is the existing Express rate-limit middleware already used on other API routes (defined in the project's middleware setup). It was applied as middleware to both routes. The owner auth check uses `sdk.authenticateRequest(req)` followed by an `openId !== OWNER_OPEN_ID` comparison. Non-authenticated requests get 401; authenticated non-owner requests get 403. This exceeds the finding's requirement (which only specified "owner auth") by adding the rate limiter as defense-in-depth.
